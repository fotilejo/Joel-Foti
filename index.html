Antes — Sistema legacy

Después — Sistema modernizado

El Problema

Sistema web legacy construido sobre bases ISIS y scripts WXIS sin documentación. La lógica de búsqueda estaba dispersa en archivos .xis y formatos ISIS, operando como silos inaccesibles para la web moderna. Sin diseño responsivo, sin exportación de datos, sin sugerencias de búsqueda.

Proceso de Ingeniería

  • Auditoría e ingeniería inversa de scripts .xis y archivos .fdt de cada base de datos

  • Extracción del tesauro → normalización a JSON → sistema de sugerencias en tiempo real integrado al buscador
  • Búsqueda avanzada multi-campo con limpieza de caracteres especiales para mejorar el recall
  • SEO técnico y security hardening en servidor Apache (headers, caché, configuración)
  • Administración del repositorio institucional en Greenstone 3: colecciones, flujos de importación y nueva interfaz

Funcionalidades Entregadas

  • Guardar y compartir búsquedas vía URLs persistentes
  • Exportación de resultados a CSV y TXT
  • Paginación moderna con selector numérico de página
  • Registros bibliográficos rediseñados, responsivos y accesibles
  • Interconexión entre Diccionarios Técnicos y Catálogo Bibliográfico

02

Data Engineering · Python · ETL

Motor de Transformación a JSON

github.com/fotilejo/data-normalizer-etl ↗

El contraste visual de la curaduría de datos: a la izquierda, la materia prima (CSV legacy) con delimitadores anómalos. A la derecha, el resultado de la transformación (JSON) con su jerarquía semántica recuperada.

Antes — CSV Legacy en crudo

Después — JSON Estructurado

El Problema

Las exportaciones masivas de sistemas bibliotecológicos legacy presentan inconsistencias críticas — conflictos de encoding, delimitadores inyectados, campos repetibles colapsados en texto plano — que impiden su migración a bases de datos modernas.

Proceso de Ingeniería

  • Script de diagnóstico exploratorio (EDA) para mapear la estructura de la base cruda
  • Limpieza estructural con Pandas: purga de nulos y estandarización snake_case
  • Transformación de cadenas planas en listas iterables para recuperar la jerarquía de metadatos
  • Exportación a JSON estructurado, listo para ingesta en bases de datos de grafos
💡 Decisiones Arquitectónicas — Human-in-the-loop

Durante el desarrollo asistido por IA, identifiqué y corregí fallas críticas en la lógica estándar de los LLMs. Forcé la reescritura de los scripts para que evadan la trampa del metadato anómalo en cabeceras legacy (sep=;) que rompía la ingesta de Pandas, garantizando un pipeline a prueba de errores humanos de exportación.

03

Knowledge Graphs · Semantic Web

Motor de Entidades y Control de Autoridades

github.com/fotilejo/Semantic-Entity-Extractor-Graph-Modeler ↗

El salto a la Web Semántica: de un catálogo estructurado pero plano (islas de información) a una topología de grafo donde documentos, autores y temas se transforman en nodos interconectados matemáticamente.

Antes — JSON Plano (materia prima)

Después — Nodos y Relaciones ( showcase autómata )

El Problema

Las bases bibliográficas tradicionales almacenan información redundante — el nombre de un autor repetido cientos de veces — creando silos que impiden motores de descubrimiento moderno, búsquedas por inferencia y arquitecturas GraphRAG para IA.

Ingeniería & Resultados

  • Extracción dinámica: Despiece algorítmico de registros planos en entidades independientes (Documentos, Autores, Temas)
  • Control de Autoridades al vuelo: Diccionarios en memoria y normalización de strings para garantizar la unicidad de cada nodo
  • Modelado Semántico: Generación de tripletas (Sujeto–Predicado–Objeto) con UUIDs trazables
  • Impacto: 2.500 registros → red semántica de 1.104 nodos únicos y 2.237 relaciones, lista para ingesta en bases de datos de grafos
💡 Decisiones Arquitectónicas — Human-in-the-loop
  • Edge cases de coautoría: Detecté que el script base trataba "Autor A; Autor B" como un solo nodo. Intervine implementando algoritmos de partición para liberar a cada coautor como entidad independiente
  • Fallas de capitalización: Identifiqué que variaciones mínimas ("Información" vs "información") duplicaban nodos. Dirigí la normalización semántica al vuelo para blindar el Control de Autoridades

04

Graph Databases · Web Semántica W3C

Motores de Grafo y Razonamiento Semántico

Bifurcación del catálogo estructurado hacia dos arquitecturas especializadas: rendimiento de búsqueda por afinidad (Property Graphs) versus rigor lógico para inferencias complejas (Web Semántica W3C).

Sprint 2.1 — Materialización y GraphRAG (Property Graphs)

Tecnologías: Neo4j · Cypher · APOC

Ingesta del catálogo bibliotecario (1.104 nodos y 2.237 relaciones) en Neo4j. Backend de alto rendimiento capaz de ejecutar consultas analíticas multi-salto, habilitando motores de recomendación por Content-Based Filtering — calculando distancia semántica mediante descriptores temáticos compartidos en la red.

Sprint 2.2 — Lógica y Ontologías · Web Semántica W3C EN DESARROLLO

Tecnologías: RDF · OWL · SPARQL · Protégé

Construcción de un Grafo de Conocimiento Histórico usando rigor lógico estricto (tripletas Sujeto–Predicado–Objeto) para que el sistema realice inferencias automatizadas sobre procesos históricos complejos — combinando formación en Historia con arquitectura semántica W3C.

💡 Resoluciones de Arquitectura Backend — Human-in-the-loop
  • Debugging silencioso en ingesta: Los scripts iniciales fallaban silenciosamente (0 relaciones creadas). Audité la lógica y detecté la discrepancia: el motor buscaba la llave relaciones mientras yo había mapeado la topología bajo enlaces. Refactoricé las consultas Cypher integrando OPTIONAL MATCH y blindaje de tipos, logrando la conexión total del grafo
  • Bypass de restricciones de entorno: Ante bloqueos de seguridad locales de APOC en Windows, reorienté la arquitectura para forzar lectura del archivo maestro directamente desde el repositorio raw en GitHub, garantizando despliegue sin fricciones
  • Algoritmo de recomendación: Diseñé y probé un algoritmo de Content-Based Filtering que supera holgadamente los límites de los JOINs tradicionales en SQL para este tipo de consulta de afinidad semántica
